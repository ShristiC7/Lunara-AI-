# Plan 2.1: Cycle Endpoints

## Goal
Establish CRUD capabilities for menstrual cycle records, strictly isolated to the authenticated `userId`.

## Tasks

<task>
<read_first>
- api/prisma/schema.prisma
</read_first>
<action>
Create Zod schema `api/src/schemas/cycle.schema.ts`.
Export `createCycleSchema` wrapping `body` requiring `startDate` (ISOString), and optional `endDate` (ISOString) where `endDate` must be >= `startDate` if available.
Export `updateCycleSchema` as partial, same rules.
</action>
<acceptance_criteria>
`cycle.schema.ts` exports `createCycleSchema` and `updateCycleSchema`.
</acceptance_criteria>
</task>

<task>
<read_first>
- api/src/schemas/cycle.schema.ts
- api/src/utils/errors.ts
</read_first>
<action>
Create `api/src/services/cycle.service.ts`.
Export class `CycleService`.
- `createCycle`: takes `userId`, `startDate`, `endDate`. Insert using prisma.
- `updateCycle`: update by `id` AND `userId` (to enforce boundary). If `startDate` and `endDate` are both present on the record, compute `cycleLength` (days difference) and update it in DB or return computed. (For schema, `cycleLength` is an Int).
- `getCycles`: takes `userId`, `page=1`, `limit=20`. use prisma `findMany` matching `userId`, order `startDate` desc, skip `(page-1)*limit`, take `limit`. Include `_count: { select: { symptoms: true } }`.
- `getCycleById`: find first by id AND userId. Include symptoms.
- `deleteCycle`: delete where id and userId match.
</action>
<acceptance_criteria>
`cycle.service.ts` contains `createCycle`, `updateCycle`, `getCycles`, `getCycleById`, and `deleteCycle` tracking `userId` in all prisma queries correctly.
</acceptance_criteria>
</task>
