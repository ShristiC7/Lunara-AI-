# Requirements: Lunara AI

**Defined:** 2026-04-25
**Core Value:** An individual woman can log her cycle and symptoms and receive accurate AI-powered predictions, contextual insights, and downloadable health reports — all from a single private account.

## v1 Requirements

Requirements for initial release. Each maps to exactly one roadmap phase.

### Authentication

- [ ] **AUTH-01**: User can register with email and password
- [ ] **AUTH-02**: User can log in and receive a JWT access token + refresh token
- [ ] **AUTH-03**: User can refresh their access token using a valid refresh token (with token rotation — old token invalidated on use)
- [ ] **AUTH-04**: User can log out (refresh token invalidated server-side)
- [ ] **AUTH-05**: User can request a password reset via email link (token expires in 1 hour)
- [ ] **AUTH-06**: User can reset their password using a valid reset token
- [ ] **AUTH-07**: All protected endpoints reject requests with invalid or expired JWT

### User Profile

- [ ] **PROF-01**: User can view their account details (email, created date)
- [ ] **PROF-02**: User can update their display name and optional bio
- [ ] **PROF-03**: User can delete their account (all associated data deleted via cascade)

### Cycle Logging

- [ ] **CYCL-01**: User can log a new menstrual cycle with a start date
- [ ] **CYCL-02**: User can update a cycle with an end date; cycle length is auto-calculated
- [ ] **CYCL-03**: User can view their full cycle history (paginated)
- [ ] **CYCL-04**: User can view details of a specific cycle (dates, length, linked symptoms)
- [ ] **CYCL-05**: User can delete a cycle (and its linked symptoms)

### Symptom Logging

- [ ] **SYMP-01**: User can log daily symptoms: mood (1–5), energy level (1–5), flow intensity (0–4), pain level (0–5), and free-text notes (max 500 chars)
- [ ] **SYMP-02**: User can update a symptom entry for a given date
- [ ] **SYMP-03**: User can view all symptom entries for a specific cycle
- [ ] **SYMP-04**: User can delete a symptom entry
- [ ] **SYMP-05**: Symptom notes are sanitized before storage (strip HTML/special chars)

### AI Cycle Prediction

- [ ] **PRED-01**: System predicts user's next period start date from their cycle history
- [ ] **PRED-02**: System estimates ovulation window (typically 14 days before predicted period)
- [ ] **PRED-03**: Prediction uses ML model (scikit-learn) when ≥3 cycles logged; falls back to rolling average when <3
- [ ] **PRED-04**: Prediction response includes a confidence level (LOW/MEDIUM/HIGH based on cycle count)
- [ ] **PRED-05**: User sees prediction with a message if data is insufficient ("Log more cycles for higher accuracy")

### AI Symptom Analysis (Insights)

- [ ] **INSG-01**: User can trigger an AI symptom analysis for their current or most recent cycle
- [ ] **INSG-02**: AI analysis is processed asynchronously via Bull queue (returns jobId + 202 immediately)
- [ ] **INSG-03**: AI returns a structured insight: pattern summary, notable symptoms, personalized recommendation
- [ ] **INSG-04**: Insights are generated by GPT-4o with structured prompt; response is validated with Pydantic/Zod
- [ ] **INSG-05**: OpenAI calls use exponential backoff retry (up to 5 attempts) for rate limit errors
- [ ] **INSG-06**: Insight results are stored in `ai_insights` table and retrievable by user
- [ ] **INSG-07**: User can view all past insights (most recent first, paginated)
- [ ] **INSG-08**: Rate limit: maximum 10 AI analysis requests per user per day (enforced in Bull queue + middleware)

### PDF Health Report

- [ ] **RPRT-01**: User can request a PDF health report for the last 3 or 6 months
- [ ] **RPRT-02**: Report generation is processed asynchronously via Bull queue (returns jobId immediately)
- [ ] **RPRT-03**: Report includes: cycle summary table, symptom trends chart (text-based), AI insights summary, and prediction for next cycle
- [ ] **RPRT-04**: Report is rendered using a Handlebars HTML template and converted to PDF via Puppeteer
- [ ] **RPRT-05**: Generated PDF is uploaded to S3 (private bucket); pre-signed URL with 1-hour expiry is stored in DB
- [ ] **RPRT-06**: User can view their report history (list of past reports with download links)
- [ ] **RPRT-07**: User can download a report via a fresh pre-signed URL (re-generated on demand)

### Email Delivery

- [ ] **MAIL-01**: Completed health report is automatically emailed to the user's registered email as an attachment
- [ ] **MAIL-02**: Password reset email is sent via Nodemailer with a secure tokenized link
- [ ] **MAIL-03**: Email sends are queue-backed via Bull with retry on SMTP failure (up to 3 attempts)
- [ ] **MAIL-04**: All emails use a consistent branded HTML template

### Background Jobs & Infrastructure

- [ ] **JOBS-01**: Bull queue for `ai-inference` jobs with configurable concurrency
- [ ] **JOBS-02**: Bull queue for `pdf-generation` jobs with configurable concurrency
- [ ] **JOBS-03**: All Bull queues have `failed` event handlers that log errors and write failure state to DB
- [ ] **JOBS-04**: Completed and permanently failed jobs are cleaned from Redis to prevent OOM
- [ ] **JOBS-05**: Job status endpoint: user can poll `GET /api/jobs/:jobId/status` for processing state

### Security & Observability

- [ ] **SECU-01**: All user-owned resource queries include `userId` in the `where` clause (no horizontal privilege escalation)
- [ ] **SECU-02**: Helmet middleware sets security headers on all responses
- [ ] **SECU-03**: Rate limiting applied globally (100 req/15min per IP) and per-user on AI endpoints
- [ ] **SECU-04**: Audit log written for every data-mutating action (create/update/delete) with userId, action, resource
- [ ] **SECU-05**: Winston structured JSON logging (no PII/health data in log payloads — IDs only)
- [ ] **SECU-06**: All environment secrets loaded from `.env`; no hardcoded credentials in source

### API Infrastructure

- [ ] **INFR-01**: `GET /health` endpoint returns service status, version, and uptime
- [ ] **INFR-02**: Global error middleware formats all errors consistently (`{ success: false, error: { code, message } }`)
- [ ] **INFR-03**: All async route handlers wrapped with asyncHandler to prevent unhandled promise rejections
- [ ] **INFR-04**: Graceful shutdown: server stops accepting requests, drains active connections, closes DB + Redis

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### Notifications

- **NOTF-01**: Email reminders: period approaching (3 days before predicted date)
- **NOTF-02**: Email reminder: log today's symptoms (configurable frequency)
- **NOTF-03**: User can configure which notification types they receive

### Analytics & Trends

- **ANAL-01**: User can view symptom trend charts across the last 6 months
- **ANAL-02**: User can identify cycle anomalies (irregularity alerts)
- **ANAL-03**: AI generates 3-month trend summary with personalized insights

### Account Management

- **ACCT-01**: User can export all their data as JSON (GDPR right to portability)
- **ACCT-02**: User can opt into anonymous aggregate research (privacy-preserving)

## Out of Scope

| Feature | Reason |
|---------|--------|
| OAuth / social login (Google, Apple) | Email/password sufficient for v1; adds auth complexity |
| Healthcare provider portal | Requires HIPAA BAAs, separate auth system — personal use only |
| Mobile app (iOS/Android) | Web API first; mobile is a separate milestone |
| Real-time push notifications (FCM/APNs) | Email-only in v1; complexity vs value |
| Wearable sync (Apple Health, Garmin) | Each requires separate OAuth + data mapping; v2 feature |
| Social features (community, sharing) | Fundamental privacy risk for health data |
| In-app AI chat (multi-turn) | Unbounded token cost; use structured insight cards instead |
| Full HIPAA compliance | BAAs + certified infra = significant cost/legal overhead; startup-grade security sufficient |
| Subscription / payments | Free in v1 |
| Custom AI model training pipeline | Training runs offline; inference-only in v1 |
